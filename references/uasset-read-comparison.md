# uasset_read Comparison Workflow

Use this reference when a user asks to compare a `.uasset` parser result against Unreal Editor truth through Unreal MCP.

## Boundary

UE MCP is an editor-visible semantic oracle. It can confirm Blueprint graphs, parent classes, variables, event nodes, pin connections, selected/open assets, logs, and scene state.

UE MCP does not prove raw `.uasset` binary serialization fidelity. Keep parser acceptance tied to parser outputs, real sample repros, and tests.

## Inputs

Collect these paths before comparing:

- Parser output report or JSON/Markdown generated by `uasset_read`.
- UE asset object path, for example `/Game/FirstPerson/Blueprints/BP_FirstPersonCharacter.BP_FirstPersonCharacter`.
- Optional source comparison files, such as paired C++ `.h` and `.cpp`.

If the user provides only a filesystem `.uasset` path, infer the UE object path from the project `Content/` relative path and verify it with Asset or Blueprint tools before relying on it.

## MCP Read Sequence

1. Call `list_toolsets` and confirm Blueprint/editor Toolsets are available.
2. Call `describe_toolset` for the Blueprint Toolset before forming arguments.
3. Call `BlueprintTools.list_graphs` for the Blueprint asset.
4. Call `BlueprintTools.get_graph` for `EventGraph` or the graph requested by the user.
5. Try `BlueprintTools.read_graph_dsl` using the graph ref returned by `get_graph`.
6. If Graph DSL is empty or rejected, call `BlueprintTools.find_nodes` with `title: ""`.
7. Pass all node refs to `BlueprintTools.get_node_infos`.
8. For a specific execution chain, call `BlueprintTools.get_connected_subgraph` from the event/input node ref.

Do not treat an empty Graph DSL result as proof that the graph is empty.

## Report Shape

Separate the report into:

- Editor semantic facts from MCP: graphs, events, variables, parent class, execution flow, data flow, pin categories, and connected nodes.
- Parser output facts: node names, decoded properties, serialized fields, warnings, partial/unknown markers, and markdown/json structure.
- C++ comparison facts when provided: class, input bindings, component setup, methods, and property names.
- Gaps: parser missing data, MCP unavailable data, and expected differences between Blueprint and C++ implementations.

Use MCP data as supporting evidence, not as a replacement for real parser acceptance.
